# 如何解决核心问题

## 1. 这道题真正难在哪里

这道题表面上是在做一个“双向翻译工具”，但真正的难点不是调用大模型，而是让模型输出的内容真正具备“跨角色沟通价值”。

题目里的核心挑战可以归纳为 3 个问题：

1. 如何让 AI 理解产品经理和开发工程师的思维模式差异？
2. 如何让 AI 在翻译时主动补充缺失信息？
3. 如何让输出结果真正有实用价值，而不是只把原文换一种说法？

这个项目的解决方式，不是依赖一个泛化的大 prompt，而是把问题拆成：

- 角色建模
- 输出约束
- 缺口信息补全
- 结构化结果
- 工程化落地

---

## 2. 如何让 AI 理解产品经理和开发工程师的思维模式差异

### 2.1 核心做法：拆成两套完全独立的 system prompt

项目没有使用一个“万能翻译 prompt”，而是为两个方向分别设计了两套 system prompt：

- `产品 -> 开发`
- `开发 -> 产品`

对应代码位置：

- [app/prompts.py](/Users/dubiao/Desktop/bb/bs/agent/app/prompts.py)

### 2.2 为什么这样做

因为产品经理和开发工程师的关注点不是“语气不同”，而是“信息选择标准不同”。

例如：

- 产品经理更关注：
  - 用户价值
  - 商业目标
  - 功能表达
  - 指标提升

- 开发工程师更关注：
  - 功能边界
  - 技术路径
  - 数据来源
  - 实时性要求
  - 工作量评估

所以 prompt 里不是简单说“请翻译给开发”，而是明确规定：

- 目标读者是谁
- 输出要优先覆盖哪些信息
- 哪些表达方式无效
- 最终必须按什么结构输出

### 2.3 具体是怎么写的

`产品 -> 开发` 的 prompt 强制模型输出：

- 需求目标
- 建议技术路径
- 数据与依赖
- 性能与实时性
- 工作量评估
- 待确认问题

`开发 -> 产品` 的 prompt 强制模型输出：

- 技术变更概要
- 用户体验影响
- 业务价值
- 成本与效率收益
- 风险与限制
- 建议沟通口径

这样模型不是在“同义改写”，而是在“换受众、换关注重点、换沟通策略”。

## 3. 如何让 AI 主动补充缺失信息

### 3.1 核心做法：明确要求模型区分“事实、推断、待确认”

很多这类题目的问题是：模型会自动补很多内容，但会把猜测写得像确定事实。

这个项目专门在 prompt 中加入了约束：

- 不要编造确定事实
- 必须区分：
  - 已知事实
  - 合理推断
  - 待确认问题

这一步很关键，因为真实工作场景里最怕的是“看起来很完整，但其实是错的”。

### 3.2 用户模板再次强化“主动补全”

除了 system prompt，用户输入还会被包装成带有明确指令的 user prompt。

例如：

- `产品 -> 开发` 时，会补一句：
  - “请主动补全开发关注点”
  - “如果信息缺失，请在待确认问题中列出”

- `开发 -> 产品` 时，会补一句：
  - “请解释成业务语言”
  - “如果业务影响无法直接确定，请明确写成推断或建议进一步确认”

对应代码位置：

- [app/prompts.py](/Users/dubiao/Desktop/bb/bs/agent/app/prompts.py)

### 3.3 这样设计的价值

这让模型具备两个重要能力：

1. 能主动把另一角色最关心、但原文没写清的内容补出来
2. 又不会因为“补充”而把不确定信息伪装成事实

这就是“有帮助”和“瞎补充”之间的区别。

## 4. 如何让输出结果真正有实用价值

### 4.1 核心做法：固定输出结构，而不是自由散文

如果输出是自由文本，往往会出现两个问题：

- 内容看起来顺，但不好扫描
- 不适合真实讨论和评审场景

所以这个项目强制模型按固定标题输出。

这样做的好处是：

- 读者能快速扫重点
- 适合需求评审、方案同步、周报沟通
- 前端可以直接把结果按 Markdown 结构化渲染

### 4.2 “实用价值”是如何体现的

#### 对产品 -> 开发

不只是把“我们要做推荐”换一种说法，而是要补出：

- 推荐算法方向
- 数据依赖
- 性能预期
- 粗粒度工作量
- 还缺哪些信息

#### 对开发 -> 产品

不只是把“QPS 提升了 30%”说得更通俗，而是要补出：

- 用户体验有什么变化
- 是否能支撑业务增长
- 是否可能降低成本
- 风险和限制是什么

这样输出的内容才能真正进入实际讨论，而不是只当成一个展示 Demo。

## 5. 工程上是怎么落地这套提示词方案的

### 5.1 方向选择

在请求体中定义了固定的翻译方向：

- `pm_to_dev`
- `dev_to_pm`

对应代码位置：

- [app/schemas.py](/Users/dubiao/Desktop/bb/bs/agent/app/schemas.py)

### 5.2 Prompt 组装

在翻译服务里通过 `build_messages()` 根据方向选择不同的 system prompt 和 user template。

对应代码位置：

- [app/translator.py](/Users/dubiao/Desktop/bb/bs/agent/app/translator.py)

这一步的作用是：

- 把“方向选择”转成“模型行为差异”
- 保证同一方向输出风格稳定

### 5.3 场景识别作为轻量增强

项目还加了一个轻量加分项：根据关键词判断输入更像：

- 需求讨论
- 技术方案
- 待判断

这个功能不是直接改写 prompt，而是用于界面提示和增强用户理解。

对应代码位置：

- [app/translator.py](/Users/dubiao/Desktop/bb/bs/agent/app/translator.py)

### 5.4 流式输出 + Markdown 渲染

模型返回结果通过 SSE 流式输出到前端。

前端一边显示“流式轨迹”，一边把累积文本做 Markdown 渲染，形成结构化预览。

这样可以同时满足：

- 生成过程可见
- 输出格式稳定
- 内容更像一份可以审阅的稿件

## 6. 这套方案为什么有效

总结起来，这个项目解决核心问题的方法不是“让模型更会说”，而是：

### 第一层：角色建模

先让模型知道自己是在替谁说话、说给谁听。

### 第二层：关注点约束

让模型知道每种角色最关心的信息维度是什么。

### 第三层：缺口补全机制

让模型在信息不足时主动补位，但不伪造事实。

### 第四层：结构化输出

让结果天然适合评审、同步和前端展示。

### 第五层：工程化实现

把提示词策略变成确定的代码路径，而不是靠临场 prompt 手工拼接。

## 7. 面试时可以怎么讲

如果在面试里解释这部分，可以用下面这段话：

> 我做的不是一个“把一句话翻译成另一种说法”的工具，而是把模型限定成两个不同岗位的沟通代理。  
> 对产品到开发，我要求它补齐技术路径、数据依赖、性能要求和工作量评估；  
> 对开发到产品，我要求它解释用户体验、业务价值、成本收益和风险。  
> 同时我在 prompt 里强制区分“已知事实、合理推断、待确认问题”，这样模型能主动补充，但不会胡乱编造。  
> 最后再通过结构化 Markdown 输出，让结果既适合流式展示，也适合真实讨论场景使用。

---

## 8. 一句话结论

这道题最核心的不是“调用大模型”，而是把大模型从“会生成文字”变成“懂角色、懂缺口、懂沟通目标”的翻译代理。
